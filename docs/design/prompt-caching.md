# Design Spec — managed-prompt-caching

**Effort:** M → revised **M**  |  **Verdict:** `sound_with_fixes`  |  **Feasible:** True  |  **Depends on:** none

## Current state
vibe has TWO caching postures today, split by backend.

1. ANTHROPIC / VERTEX path (explicit, already implemented): `AnthropicAdapter._build_system_blocks` (vibe/core/llm/backend/anthropic.py:369-378) stamps `cache_control: {type: ephemeral}` on the system text block, and `_add_cache_control_to_last_user_message` (anthropic.py:380-392) stamps an ephemeral breakpoint on the final user/tool_result block. Beta header `prompt-caching-2024-07-31` is sent (anthropic.py:343-348, 496). Anthropic accounting is parsed correctly: `prompt_tokens = input_tokens + cache_creation_input_tokens + cache_read_input_tokens` (anthropic.py:181-186, 316-320, 542-546). NOTE: it does NOT populate `LLMUsage.cached_tokens` — that field stays 0 on the Anthropic path, so telemetry can't see Anthropic cache hits.

2. GENERIC / OpenAI-compatible path (auto-cache only): `OpenAIAdapter.build_payload` (generic.py:36-70) emits a plain Chat Completions body with no cache hints. It relies entirely on provider auto-caching (OpenAI, DeepSeek, Fireworks, GLM/Zhipu, Together, etc. cache prefixes automatically). It DOES now read cache hits back: `parse_response` (generic.py:190-206) reads `usage.prompt_tokens_details.cached_tokens` (OpenAI shape) OR top-level `usage.cached_tokens` (DeepSeek shape) into `LLMUsage.cached_tokens`.

`extra_body` plumbing exists end-to-end and is honored ONLY by the generic + reasoning adapters: `OpenAIAdapter.build_payload` merges it last via `payload.update(extra_body)` (generic.py:65-68); `ReasoningAdapter` does the same (reasoning_adapter.py:151-152); anthropic/mistral/vertex/openai-responses adapters explicitly `del extra_body` (anthropic.py:471, mistral.py:300/386, vertex.py:83, openai_responses.py:561). Config source for extra_body today: per-call only — `SafetyJudgeConfig.extra_body` (_settings.py:238) passed by safety_judge.py:140. NEITHER `ModelConfig` (_settings.py:545-555) NOR `ProviderConfig` (_settings.py:273-284) has an `extra_body` field, and the main turn loop does NOT pass extra_body: `_chat` (agent_loop.py:1854-1864) and `_chat_streaming` (agent_loop.py:1931-1941) call `backend.complete(...)`/`complete_streaming(...)` without an `extra_body` kwarg, so the main agent's generic-path requests currently send none.

`cached_tokens` is plumbed into `LLMUsage` (types.py:354-363, with `__add__` summing it) but is consumed NOWHERE: `_update_stats` (agent_loop.py:1985-1993) reads only prompt/completion tokens; `AgentStats` (types.py:47-139) has no cache fields; `session_cost` (types.py:103-116) explicitly ignores caching ("worst-case", comment line 107).

Prefix-stability facts that matter for auto-cache: messages are sent in a stable order via `_messages_for_backend` (agent_loop.py:1804-1812, identity passthrough unless stripping images for a non-image model). System prompt is built once per session and embeds today's date (system_prompt.py:192-194) and cwd (line 65, 419) — these are stable WITHIN a session, so the cacheable prefix is not poisoned mid-session. Tools are serialized deterministically from `tool.model_dump(exclude_none=True)` (generic.py:54). Risk to prefix stability: middleware `INJECT_MESSAGE` reminders (middleware.py:147,193-233) append messages but at the tail, not the head, so they don't break the stable prefix.

## Target design
Be honest up front: for the dominant generic-path providers (OpenAI, DeepSeek, GLM, Fireworks, Together, Groq), prefix auto-caching is fully automatic and requires NO cache hints — explicit markers there are either ignored or a no-op. The high-value work is therefore (a) protecting and maximizing the auto-cache prefix that already works, (b) a thin, OPT-IN escape hatch to inject provider-specific cache markers for the minority of OpenAI-compatible providers that expose them, and (c) telemetry so the operator can actually decide whether (b) is ever worth enabling. We do NOT build a generic "managed breakpoint planner" for the OpenAI path because no widely-used OpenAI-compatible provider accepts Anthropic-style per-message breakpoints over /chat/completions; that would be speculative complexity.

Three components:

A) STABLE CACHEABLE-PREFIX GUARANTEE (the load-bearing change; works for all auto-cache providers).
The cache prefix on the generic path is `[system] + [tools] + [history...]`. Auto-cache hits require this prefix to be BYTE-IDENTICAL turn over turn. Today it mostly is, but there are three latent poisoners to lock down:
  1. Tool ordering / serialization: ensure `payload["tools"]` is built from a stable-ordered tool list with deterministic JSON (already true via model_dump, but assert ordering is stable across turns — tools come from `format_handler.get_available_tools`, which is order-stable per session).
  2. Volatile content placement: any per-turn injected reminder / environment-refresh must be appended at the TAIL (last user/tool message), never spliced into the system prompt or early history. This is already the case for middleware injects; the design codifies it as an invariant and adds a guard. The "volatile suffix last" rule: system + tools + all-but-last message form the stable prefix; the final user/tool block is the volatile suffix.
  3. Date/cwd in system prompt: stable within a session (built once). Across sessions the date flip invalidates yesterday's cache — acceptable, providers re-warm in one request. No change needed; document it.
This component is pure invariant-enforcement + a test; it makes auto-cache hit rates as high as the provider allows with zero protocol changes.

B) OPTIONAL PROVIDER-SPECIFIC CACHE MARKERS via the existing extra_body mechanism (opt-in, no new protocol).
Add `ProviderConfig.cache: ProviderCacheConfig` and `ModelConfig`-level override. When `cache.mode == "explicit"`, the generic backend asks a small `CacheHintStrategy` (keyed by `cache.style`) to produce a body fragment that is merged exactly like extra_body. Supported styles:
  - `"off"` (default): emit nothing. Correct for OpenAI/DeepSeek/GLM/Together/Groq auto-cache.
  - `"anthropic-compat"`: for OpenAI-compatible gateways that proxy to Anthropic and accept `cache_control` on message content (e.g. some OpenRouter routes). Strategy converts the last system message + last user message into structured `content` arrays with `cache_control: {type: ephemeral}` (mirrors the anthropic.py:369-392 pattern but on the OpenAI message shape).
  - `"passthrough"`: merge a literal operator-supplied `cache.extra_body` fragment verbatim (escape hatch for any future provider knob, e.g. `{"prompt_cache_key": "..."}` for OpenAI's explicit cache routing, or `{"cache": {"ttl": "1h"}}`).
The strategy output is merged into the payload through the SAME code path as extra_body (generic.py:65-68), so no new merge logic and provider adapters that `del extra_body` remain unaffected.

C) MEASUREMENT / TELEMETRY using cached_tokens (decides whether B is ever worth it).
Add cache accounting to `AgentStats` and surface a cache-hit ratio. First make the Anthropic path populate `LLMUsage.cached_tokens` (set it to `cache_read_input_tokens` in the three anthropic.py parse sites) so telemetry is backend-uniform. Then `_update_stats` accumulates `session_cached_tokens` and computes `cache_hit_ratio = cached / prompt`. Expose it in the status line and session telemetry. Operator workflow: run with `cache.mode == "off"` (auto), read the hit ratio; only flip a provider to `"explicit"`/`"passthrough"` if its auto-cache ratio is poor AND the provider documents an explicit knob. This keeps the feature honest and self-justifying.

## Config

- `providers[].cache.mode` (enum(off|explicit), default `off`) — Master switch. 'off' = rely on provider auto-cache (correct for OpenAI/DeepSeek/GLM/Together/Groq). 'explicit' = emit cache hints via the style strategy.
- `providers[].cache.style` (enum(off|anthropic-compat|passthrough), default `off`) — Which hint strategy to use when mode=explicit. anthropic-compat = cache_control tags on message content (for Anthropic-proxying OpenAI gateways). passthrough = merge literal extra_body fragment.
- `providers[].cache.extra_body` (dict, default `{}`) — Literal request-body fragment merged when style=passthrough (e.g. {"cache": {"ttl": "1h"}}). Escape hatch for any provider-specific cache knob.
- `providers[].cache.cache_key` (str|null, default `null`) — Optional stable cache routing key (e.g. OpenAI prompt_cache_key). Injected as {"prompt_cache_key": <value>} under passthrough/explicit. Should be stable per session/agent-role to maximize affinity.
- `models[].cache` (ProviderCacheConfig|null, default `null`) — Optional per-model override of the provider cache config; null = inherit provider.cache.

## Algorithm
 1. Resolve cache config at request build time: in OpenAIAdapter.prepare_request, effective = model.cache or provider.cache (provider arg already available; model-level needs model passed or resolved by caller).
 2. If effective.mode == 'off' OR effective.style == 'off': skip hinting entirely (the common, correct path). Auto-cache does the work.
 3. Else call build_cache_hint(effective, converted_messages): dispatch on effective.style.
 4. style=='passthrough': fragment = deepcopy(effective.extra_body); if effective.cache_key: fragment.setdefault('prompt_cache_key', effective.cache_key); return fragment.
 5. style=='anthropic-compat': locate last system message and last user/tool message in converted_messages; rewrite each message's string content into [{'type':'text','text':<content>,'cache_control':{'type':'ephemeral'}}]; only tag at most 2 breakpoints (system prefix + volatile suffix) to mirror anthropic.py; return {} (mutation is in-place on messages) OR return a structured override — choose in-place mutation to match existing OpenAI message shaping.
 6. Merge: caller_extra_body = dict(extra_body or {}); for k,v in hint.items(): caller_extra_body.setdefault(k, v) (caller/extra_body wins). Pass merged dict into build_payload so payload.update(extra_body) (generic.py:68) applies it last.
 7. STABLE-PREFIX INVARIANT (always, independent of mode): the request builder must never insert per-turn volatile text into the system prompt or any non-final message. Add an assertion/guard test that system+tools+messages[:-1] are byte-identical across two consecutive turns given identical history.
 8. On response parse (already present): generic.py:197-204 reads cached_tokens; anthropic.py adds cache_read_input_tokens -> cached_tokens (new).
 9. _update_stats accumulates session_cached_tokens and last_turn_cached_tokens; cache_hit_ratio computed on read.
 10. Operator decision loop: observe cache_hit_ratio under mode=off; only enable explicit/passthrough for a provider whose auto ratio is low and that documents an explicit knob.

## Edge cases
- Provider auto-caches AND operator also sets passthrough markers the provider ignores: harmless no-op, markers are extra keys the provider drops. Document that explicit is only meaningful for providers that document the knob.
- anthropic-compat against a provider that does NOT accept cache_control in OpenAI messages: provider may 400 on the unexpected content shape. Mitigation: anthropic-compat must be opt-in per provider; ship it OFF by default and warn in docs that it only suits Anthropic-proxying gateways.
- Date flip mid-session (observed: this run crossed 2026-06-19 -> 2026-06-20): system prompt is built once at session start, so the in-session prefix stays stable; only a NEW session after midnight re-warms cache. No action needed but must be noted so nobody 'fixes' the date to be dynamic per-turn (which would destroy the prefix).
- Image-stripping for non-image models (_messages_for_backend, agent_loop.py:1809-1811) rewrites message content and changes the prefix bytes vs an image-capable model — but it's stable per (model, history), so auto-cache still hits within a session. No change.
- extra_body key collision between caller (e.g. safety_judge GLM thinking-disable) and a cache hint: caller wins (setdefault), so cache hint never overrides an explicit operator/feature body.
- cache_key reused across two concurrently running agents with DIFFERENT prefixes: provider may serve a stale/foreign prefix or just miss. Recommend deriving cache_key per session/agent-role, never a global constant.
- Streaming vs non-streaming: cached_tokens arrives in the final usage chunk for OpenAI-style streams; generic.py parse already handles usage on any chunk and agent_loop sums via usage += (agent_loop.py:1957). Verify the provider sends prompt_tokens_details in the stream's final usage event.
- Provider returns cached_tokens > prompt_tokens (malformed): clamp cached = min(cached, prompt_tokens) when computing ratio to avoid >100%.

## Test plan
- Unit: build_cache_hint('off') returns None; ('passthrough', extra_body={'cache':{'ttl':'1h'}}, cache_key='k') returns {'cache':{'ttl':'1h'},'prompt_cache_key':'k'}.
- Unit: anthropic-compat rewrites only last system + last user message content into cache_control-tagged arrays, leaves middle history untouched, tags <=2 breakpoints.
- Unit: prepare_request merge precedence — caller extra_body key beats cache-hint key (setdefault semantics).
- Invariant test: build two payloads from identical message history on consecutive 'turns'; assert system block + tools + messages[:-1] serialize byte-identically (locks the stable-prefix guarantee).
- Regression: with cache.mode=off (default), generic payload is byte-identical to today's (no cache keys leak in) — protects the auto-cache path.
- Parse test: generic.py parse with OpenAI shape (prompt_tokens_details.cached_tokens) and DeepSeek shape (top-level cached_tokens) both populate LLMUsage.cached_tokens (already covered; extend).
- Parse test: anthropic.py parse_response/_parse_message_start now set cached_tokens = cache_read_input_tokens; assert prompt_tokens still = input+create+read (no double count).
- Stats test: _update_stats accumulates session_cached_tokens; cache_hit_ratio = cached/prompt with div-by-zero guard and clamp at 1.0.
- Adapter-isolation test: anthropic/mistral/vertex/openai_responses adapters still ignore extra_body (del) and are unaffected by cache config (no cache key in their payloads).
- Integration smoke (mocked HTTP): two sequential generic completions over a growing history report rising cached_tokens from a fake provider that echoes prefix-length as cached.

## Risks
- Over-engineering risk: for the providers vibe actually targets, auto-cache already works and explicit hints add zero value. If the explicit path is built but never beneficial, it is dead config surface. Mitigation: ship mode=off default, gate explicit behind telemetry-proven need, document honestly.
- anthropic-compat content-shape rewrite can 400 a provider that doesn't expect cache_control in OpenAI messages — a correctness regression if defaulted on. Mitigation: opt-in only, off by default, clear docs.
- Prefix-stability is the real lever and is fragile: any future feature that injects per-turn text into the system prompt or early history silently destroys auto-cache hit rates with no error. Mitigation: the invariant test + a code comment at the inject sites (middleware.py) and system_prompt build.
- Double-counting cached tokens into cost: cache reads are cheaper, but session_cost (types.py:103-116) bills all prompt_tokens at full input price. Surfacing cache_hit_ratio without correcting cost may mislead. Mitigation: keep cost as documented worst-case; optionally add a separate effective-cost estimate using cached_tokens * cache_read_discount — but only if a per-provider discount is configured (out of scope, note as follow-up).
- cache_key misuse (global constant) can cause cross-agent cache bleed/misses. Mitigation: doc + recommend per-session derivation.
- Streaming providers that omit prompt_tokens_details in the final usage event yield cached_tokens=0 -> telemetry under-reports; not a correctness bug but can cause a wrong 'auto-cache is bad' conclusion. Mitigation: note provider-dependence in docs.
