# Design Spec — max-output-escalation

**Effort:** M → revised **M**  |  **Verdict:** `sound_with_fixes`  |  **Feasible:** True  |  **Depends on:** none

## Current state
When the LLM truncates a response (or the requested `max_tokens` exceeds the model cap), the backend (orchestral_runtime / generic / mistral) returns HTTP 422 whose body contains `finish_reason=length` or `max_tokens_exceeded`. `BackendError.is_response_too_long` (vibe/core/llm/exceptions.py:77-81, substrings at :27) flags it. In `_chat` (vibe/core/agent_loop.py:1893-1894) and `_chat_streaming` (:1976-1977) this is converted to a terminal `ResponseTooLongError` (vibe/core/types.py:618-624) and propagated up through `_perform_llm_turn` (vibe/core/agent_loop.py:1227) into `_conversation_loop` (:1132-1164), where there is NO except clause for it — so it bubbles out of `act()` and kills the turn. By contrast `ContextTooLongError` (:1137-1152, self-heal compact+retry once) and `RateLimitError` (:1153-1164, fail over to fallback model + retry) are already self-healed in that same try/except, giving an exact pattern to mirror. Today `_chat`/`_chat_streaming` are always called with `max_tokens=None` (callers `_get_assistant_event` :1305 and `_stream_assistant_events` :1281 pass nothing), so the effective output cap is the backend default: anthropic uses `DEFAULT_MAX_TOKENS=8192` / `DEFAULT_ADAPTIVE_MAX_TOKENS=32768` (vibe/core/llm/backend/anthropic.py:349-350,406-419); generic/mistral omit the field entirely and let the server decide (generic.py:61-62). There is no per-model or global output-token cap in config (ModelConfig at vibe/core/config/_settings.py:545-555 has temperature/prices/thinking/auto_compact_threshold but no max_tokens; VibeConfigSchema at vibe/core/config/vibe_schema.py:52-85 likewise). Instance state for the existing self-heal mechanisms (`_fallback_model_override`, `_tried_fallback_aliases`) is initialized at agent_loop.py:381-382.

## Target design
On `ResponseTooLongError`, `_conversation_loop` retries the same turn with a progressively larger `max_tokens`, mirroring the `ContextTooLongError`/`RateLimitError` self-heal arms, up to a bounded number of attempts and a hard cap, then re-raises if still failing.

Mechanism: add an instance field `self._max_output_override: int | None = None` (next to `_fallback_model_override` at agent_loop.py:381) and an attempt counter `self._response_too_long_attempts: int = 0`. `_chat` and `_chat_streaming` start by resolving `max_tokens = max_tokens if max_tokens is not None else self._max_output_override` so the override flows to `backend.complete(...)` without touching the call signatures of `_get_assistant_event`/`_stream_assistant_events` (which keep passing nothing). When the override is None the existing backend defaults apply, so behavior is unchanged until an escalation actually fires.

A new config block `max_output_escalation` (top-level on VibeConfigSchema, with per-model override fields on ModelConfig) supplies: `enabled`, `base` (the max_tokens to start escalation from when no override is active yet), `factor` (multiplier per attempt, default 2x), `cap` (absolute ceiling), and `max_attempts`. On the first `ResponseTooLongError` of a turn the loop sets the override to `min(base*factor, cap)` (base defaults to the backend's known default so the first retry is a genuine increase), then `continue`s the `while` loop to re-run `_perform_llm_turn`. Each subsequent failure multiplies by `factor`, clamped to `cap`. After `max_attempts` failures, or once the override already equals `cap` and still fails, re-raise `ResponseTooLongError` (terminal, same UX as today). The override and counter reset at the top of each new user turn (in `_conversation_loop` before the while loop) so escalation does not leak across turns; success also resets them so a later turn starts fresh.

Interaction with streaming: identical control flow. In the non-streaming path `backend.complete` raises before any message is appended, so `self.messages` is clean for retry. In streaming, the 422 is raised by the server before/at the start of the stream (it is a request-level rejection of the `max_tokens`/length condition, not a mid-stream content event), and `_chat_streaming` only appends `chunk_agg.message` AFTER the async-for completes (agent_loop.py:1965) — so on the error path no partial assistant message is left in `self.messages`. The only streaming-specific wrinkle is that partial `AssistantEvent`/`ReasoningEvent` chunks may already have been emitted to the UI before the error; on retry the model regenerates from scratch, so the UI must treat the in-progress assistant message as superseded (handled by emitting under a fresh message_id on the retried turn, which already happens since `_stream_assistant_events` derives message_id per call).

## Config

- `max_output_escalation.enabled` (bool, default `true`) — Master switch. When false, ResponseTooLongError stays terminal (current behavior).
- `max_output_escalation.base` (int, default `8192`) — Starting max_tokens used to compute the first escalation when no override is active. Should match/exceed the backend default (anthropic DEFAULT_MAX_TOKENS=8192) so the first retry is a real increase.
- `max_output_escalation.factor` (float, default `2.0`) — Multiplier applied to max_tokens each attempt (2x mirrors Claude Code).
- `max_output_escalation.cap` (int, default `65536`) — Absolute ceiling for escalated max_tokens; never exceed the model's real output limit. Per-model override via ModelConfig.max_output_tokens.
- `max_output_escalation.max_attempts` (int, default `3`) — Number of escalation retries before surfacing ResponseTooLongError (Claude Code uses up to 3).
- `models[].max_output_tokens` (int | None, default `null`) — Optional per-model output cap; when set, overrides global `cap` (and can seed `base`) so escalation respects each model's true limit.

## Algorithm
 1. At AgentLoop.__init__ (~:381): set self._max_output_override = None and self._response_too_long_attempts = 0.
 2. In _conversation_loop, before the `while not should_break_loop` loop (~:1112): reset self._max_output_override = None and self._response_too_long_attempts = 0 so escalation is scoped to this user turn.
 3. In _chat (~:1852) and _chat_streaming (~:1927): resolve effective = max_tokens if max_tokens is not None else self._max_output_override; pass effective into backend.complete/complete_streaming. When effective is None backend defaults apply (unchanged behavior).
 4. _chat/_chat_streaming convert backend 422 with finish_reason=length/max_tokens_exceeded into ResponseTooLongError exactly as today (:1893-1894 / :1976-1977) — no change there.
 5. In _conversation_loop try/except, add `except ResponseTooLongError:` after the RateLimitError arm (after :1164): call nxt = self._escalate_max_output(); if nxt is None: raise; else log.warning('Response truncated; retrying turn with max_tokens=%d', nxt) and `continue`.
 6. _escalate_max_output(): if not config.max_output_escalation.enabled -> return None. Increment self._response_too_long_attempts; if attempts > max_attempts -> return None. Compute current = self._max_output_override or escalation.base (or the active model's resolved default). Compute cap = active_model.max_output_tokens or escalation.cap. next_val = min(int(current * escalation.factor), cap). If next_val <= (self._max_output_override or 0) (i.e. already pinned at cap, no further growth) -> return None. Set self._max_output_override = next_val; return next_val.
 7. On a successful turn (no exception), self._max_output_override and counter remain but are reset at the next user turn's top — so escalation state never leaks across user turns. Optionally also reset on success inside the loop for tidiness.
 8. If escalation exhausts (returns None), ResponseTooLongError propagates out of act() exactly as it does today, preserving the existing terminal UX.

## Edge cases
- Streaming partial output already shown to user: on retry the model regenerates; the retried turn uses a fresh message_id (derived per _stream_assistant_events call) so the UI replaces rather than appends. Confirm the observer/UI treats a new assistant message_id as superseding the truncated in-progress one.
- max_tokens_exceeded vs finish_reason=length: both map to is_response_too_long, but max_tokens_exceeded means the REQUEST cap exceeded the model limit — escalating further is futile. The cap (escalation.cap / model.max_output_tokens) must be set to the model's true ceiling so _escalate_max_output returns None (pinned at cap) instead of looping uselessly. The `next_val <= current` guard handles this.
- Interaction with thinking/adaptive (anthropic): with thinking on, the default is DEFAULT_ADAPTIVE_MAX_TOKENS=32768; if `base`=8192 the first escalation (16384) could be LOWER than the model's effective default and might not raise the cap. base should account for adaptive default, or escalation should start from max(base, backend_default). Document/seed base per model.
- Interaction with ContextTooLong + escalation: a higher max_tokens reserves more output budget and can shrink the input budget, occasionally turning a ResponseTooLong into a ContextTooLong on retry — that is then handled by the existing ContextTooLongError arm (compact once). Both self-heal arms coexist in the same try/except.
- fallback model switch mid-escalation: if a RateLimitError fires during an escalated turn and switches models, the new model may have a different output limit; _escalate_max_output reads active_model.max_output_tokens each call so it re-clamps to the new model's cap.
- Compaction turns / utility _chat call at :2168 (summary): that call passes its own max_tokens explicitly (or None) — ensure the override does not silently inflate the compaction summary call; gate the override-resolution so it only applies to main-turn _chat, OR accept it (cheap). Recommend: the summary _chat at :2168 passes an explicit max_tokens so the `is not None` branch wins and the override is ignored.
- enabled=false or max_attempts=0: behavior is identical to today (terminal error), giving a clean opt-out.
- Non-422 RuntimeError wrapping: _is_response_too_long_error already unwraps RuntimeError(__cause__=BackendError); the except clause catches the typed ResponseTooLongError raised at :1894/:1977, so unwrapping is already handled upstream.

## Test plan
- Mirror tests/core/test_context_overflow_recovery.py. test_response_too_long_escalates_and_retries: monkeypatch loop._perform_llm_turn to raise ResponseTooLongError on attempt 1 then succeed; assert it retried once, that self._max_output_override was set to base*factor, and that the turn completed.
- test_response_too_long_escalates_multiple_times: raise on attempts 1..N-1, succeed on N; assert override grew geometrically (8192->16384->32768) clamped at cap and that _chat received the escalated max_tokens.
- test_response_too_long_exhausts_and_surfaces: always raise; assert exactly max_attempts retries then pytest.raises(ResponseTooLongError).
- test_escalation_clamped_to_cap: set cap below base*factor; assert override pinned at cap and that once pinned a further failure surfaces the error (no infinite loop).
- test_escalation_disabled_is_terminal: enabled=false; assert first ResponseTooLongError propagates with zero retries.
- test_override_resets_between_turns: run a turn that escalates then succeeds; start a second user turn; assert self._max_output_override is None and the second turn's _chat uses backend default.
- test_chat_passes_override_to_backend: set self._max_output_override and assert backend.complete was called with that max_tokens (both _chat and _chat_streaming).
- test_streaming_partial_then_retry_uses_fresh_message_id: simulate partial AssistantEvents before a ResponseTooLongError; assert the retried turn emits a new message_id.
- Config tests: MaxOutputEscalationConfig defaults parse; per-model max_output_tokens overrides global cap; shallow-merge of project-over-user config preserves partial overrides (consistent with recent config merge fix in commit 9ae8766).

## Risks
- Escalating max_tokens raises input/output budget pressure and can flip ResponseTooLong into ContextTooLong; mitigated because the ContextTooLong arm already self-heals, but worst case both fire and the turn compacts then retries, adding latency.
- If `cap` is set above the model's real output limit, every escalation attempt 422s with max_tokens_exceeded and burns all max_attempts before surfacing — wasteful API calls and latency. Mitigation: default cap conservative; encourage per-model max_output_tokens.
- Cost/latency: up to max_attempts extra full LLM calls per truncation, each larger. 2x growth bounds it, but a pathological loop that always truncates costs N calls per turn. max_attempts default of 3 bounds blast radius.
- Streaming UX: regenerating after partial output may show the user a flicker / duplicated-then-replaced assistant message if the UI does not key on message_id. Needs verification against the observer layer.
- The summary/compaction _chat call (agent_loop.py:2168) could inadvertently inherit the override if it passes max_tokens=None; must ensure it passes an explicit value or that override is scoped to main-turn calls only.
- Provider variance: generic/mistral backends omit max_tokens when None and may have server-side defaults different from anthropic's; `base` must be chosen so the first escalation is actually larger than each backend's default, else the first retry is a no-op that just burns an attempt.

**Wrong assumptions:**
- The spec's edge_case and risk claim that the compaction/summary _chat at line 2168 'passes an explicit max_tokens so the is not None branch wins and the override is ignored' is FALSE. The call at 2168 is `await self._chat(model_override=self.config.get_compaction_model())` — it passes NO max_tokens, so it defaults to None. Under the proposed resolution `max_tokens = max_tokens if max_tokens is not None else self._max_output_override`, this compaction call WOULD inherit a live escalation override. Since compaction can run inside the same user turn (e.g. ContextTooLong arm fires after escalation raised the override, or a /compact mid-turn), the override can silently inflate the summary call. This must be fixed by gating the override to main-turn calls or passing an explicit max_tokens at 2168.
- The spec proposes `Annotated[MaxOutputEscalationConfig, WithShallowMerge()]`, but every existing nested-config block on VibeConfigSchema (project_context:270, session_logging:273, experiments:276, worktree:279, safety_judge:282) uses WithReplaceMerge(), NOT WithShallowMerge(). WithShallowMerge maps to MergeStrategy.MERGE which raises TypeError unless both operands are dicts (vibe/core/utils/merge.py:105-107). It happens to work here because builder.py merges raw layer dicts BEFORE constructing the pydantic model (builder.py:70-111, values are raw TOML/JSON dicts), so the claim is technically valid — but it deviates from the established sibling pattern and the spec presents it as the norm.
- The spec assumes `_escalate_max_output` can read 'the active model's max_output_tokens' uniformly, but the active-model resolution differs between paths: `_chat` uses `model_override or self._fallback_model_override or get_active_model()` (1826-1830) while `_chat_streaming` uses ONLY `self.config.get_active_model()` (1905) and ignores `_fallback_model_override` entirely. So in streaming mode the fallback override is not even honored for model selection, making the 're-clamps to the new model's cap after a RateLimit fallback switch' edge-case partly moot for streaming. The helper must replicate whichever resolution it intends to clamp against.

**Missing pieces:**
- The algorithm computes the first escalation from `escalation.base` (default 8192) but does not start from max(base, backend_default). For anthropic with thinking ON the effective default is DEFAULT_ADAPTIVE_MAX_TOKENS=32768, so the first escalation to 16384 is a REGRESSION below the actual default and the second to 32768 merely re-reaches the default — burning 2 of 3 attempts on no real increase. The spec flags this in edge_cases but the algorithm body does not implement the fix; the implementation must seed the starting point from the resolved backend/model default, not a flat `base`.
- No mechanism is specified to read the backend's effective default (DEFAULT_MAX_TOKENS vs DEFAULT_ADAPTIVE_MAX_TOKENS depending on thinking) into the escalation start. This default lives on the anthropic backend class and is thinking-dependent; agent_loop has no current accessor for it. Either ModelConfig.max_output_tokens must be seeded per model or a backend-default accessor added.
- The reset-on-new-user-turn point (~line 1112, before `while not should_break_loop`) is correct, but the spec does not account for the `_run_compaction`/utility-_chat reentry: resetting only at user-turn top means a mid-turn escalation override is still live during any compaction triggered later in the same turn. Pairs with the 2168 bug above.
- ResponseTooLongError carries only (provider, model) (types.py:618-624); enriching it to mention the attempted cap (spec says 'optional') would require a signature change at both raise sites (1894, 1977) — fine to skip, but the test 'assert it surfaces' should not assert on message text.

**Corrections to fold in:**
- Fix the compaction leak: change the resolution so the override applies only to main-turn calls. Simplest: at agent_loop.py:2168 pass an explicit `max_tokens=<value or None-sentinel>` OR have `_chat`/`_chat_streaming` only consult `self._max_output_override` when no `model_override` is set (the compaction call always sets `model_override`, so `if model_override is None and max_tokens is None: max_tokens = self._max_output_override`). This cleanly excludes the compaction summary call.
- Seed the escalation starting point from the model's real default: compute `current = self._max_output_override or model.max_output_tokens or backend_default_for(active_model)` where backend_default accounts for thinking (32768 when adaptive/anthropic-thinking-on, else 8192), instead of a flat `escalation.base`. Document that `base` is a floor, and use `max(base, backend_default)` for the first step.
- Use WithReplaceMerge() to match every sibling nested-config block, OR explicitly justify WithShallowMerge() as an intentional improvement (partial-override preservation per commit 9ae8766). Either is correct; pick one consistently.
- Have `_escalate_max_output` resolve the active model the same way the path that raised did (mirror `_chat`'s `_fallback_model_override or get_active_model()`); note that streaming ignores the fallback override, so clamping in streaming should use `get_active_model()`.
- Add the cap=already-at-cap guard exactly as written (`next_val <= (self._max_output_override or 0) -> return None`) to prevent an infinite loop when max_tokens_exceeded means the request cap already exceeds the model ceiling — verified this is the right termination condition since is_response_too_long covers both finish_reason=length and max_tokens_exceeded.
