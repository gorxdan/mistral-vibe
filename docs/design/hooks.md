# Design Spec — hooks-extensibility

**Effort:** XL → revised **XL**  |  **Verdict:** `sound_with_fixes`  |  **Feasible:** True  |  **Depends on:** none

## Current state
vibe has 6 hook events (`HookType` enum, vibe/core/hooks/models.py:20-26): POST_AGENT_TURN, BEFORE_TOOL, AFTER_TOOL, TEAMMATE_IDLE, TASK_CREATED, TASK_COMPLETED. Each has an Invocation subclass of `HookSessionContext` (models.py:89-148) and is built by `build_invocation` (models.py:151-231). Per-type semantics live in `HookHandler` subclasses (`_post_agent_turn.py`, `_before_tool.py`, `_after_tool.py`, `_team_lifecycle.py`) registered in `_HANDLERS` (manager.py:39-46). `HooksManager.run` (manager.py:162-204) selects matching hooks, runs each as a subprocess (executor.py: stdin = `invocation.model_dump_json()`, stdout = optional JSON `HookStructuredResponse`), and streams `_HookYield` events.

Fire points are all in `AgentLoopHooksMixin` (vibe/core/agent_loop_hooks.py): `_run_before_tool_hooks`/`_run_before_tool_pipeline` (130-273), `_run_after_tool_hooks`/`_run_after_tool_and_finalize` (145-229), `_run_post_agent_turn_hooks`/`_dispatch_post_turn_hooks` (118-128, 417-436). `_hook_session_context` (103-112) builds the shared context. The loop drives them: post-agent-turn fires at agent_loop.py:1179-1185 (only when `should_break_loop`, i.e. the assistant stopped calling tools); before/after-tool fire inside `_handle_tool_calls` reached from `_perform_llm_turn` (agent_loop.py:1227-1252, before-tool pipeline at 1458). `reset_retry_count` fires once per user turn at agent_loop.py:1109-1110.

Config: hooks are gated by `enable_experimental_hooks` (vibe_schema.py:231, _settings.py:733) and loaded by `load_hooks_from_fs` (hooks/config.py:77-100) from `HarnessFilesManager.hook_files` (harness_files/_harness_manager.py:78-84 = `<root>/.vibe/hooks.toml` per project root + `~/.vibe/hooks.toml`). The result is threaded into `AgentLoop.__init__` as `hook_config_result` (agent_loop.py:419-426) from cli.py:325 / acp_agent_loop.py:801,1066 / programmatic via run_programmatic.

Extensibility config keys already exist and are merged with explicit strategies (vibe_schema.py): `tool_paths` (91, WithConcatMerge), `mcp_servers` (115, WithUnionMerge merge_key=name), `agent_paths` (127, WithConcatMerge), `skill_paths` (165, WithConcatMerge), `workflow_paths` (189). Consumers: agents/manager.py:99-101, skills/manager.py:68, tools/manager.py:104 and 250/298-303 (mcp_servers). There is NO plugin/bundle abstraction — each path list is configured and discovered independently. Session lifecycle: `emit_new_session_telemetry`/`emit_session_closed_telemetry` already bracket a run (programmatic.py:94,124; `_reset_session` at agent_loop.py:2048-2059 cycles them on /compact and /clear). `compact()` is at agent_loop.py:2141; `_run_compaction` wrapper at 941-982.

## Target design
Two independent deliverables, prioritized separately.

PART A — New hook events. Extend `HookType` with six values and add matching Invocation classes + `build_invocation` cases + `_HANDLERS` registrations + fire points. All six are NOTIFICATION-or-GATE events that reuse the existing subprocess/structured-response machinery; only two (UserPromptSubmit, Stop) need new gating semantics, the rest reuse a generic notification handler.

1. SessionStart (effort S) — fires once when a session begins, before the first user prompt is processed. Payload: source ("startup"|"resume"|"clear"|"compact"), plus session context. Handler: notification + optional `additional_context` injected as a system/user preamble (Claude Code parity — SessionStart can add context). Fire point: a new `_run_session_start_hooks()` called from the spots that already emit `emit_new_session_telemetry` (programmatic.py:93-94, and the interactive equivalent) and from `_reset_session` (agent_loop.py:2048-2059) with source="clear"/"compact". Returned `additional_context` is appended to `self._pending_injected_messages`.

2. SessionEnd (effort S) — fires once when a session terminates/resets. Payload: reason ("exit"|"clear"|"compact"|"logout"). Notification only (no gate, no injection — process is ending). Fire point: alongside `emit_session_closed_telemetry` (programmatic.py:124) and in `_reset_session`. Because the parent may be exiting, dispatch must be best-effort and time-bounded.

3. UserPromptSubmit (effort M) — fires at the TOP of `_conversation_loop` right after the user message is appended and the `UserMessageEvent` is yielded (agent_loop.py:1095-1110), BEFORE any LLM turn. Payload: prompt text, message_id, images-present flag. GATE semantics (new handler): `decision: "deny"` blocks the prompt (loop returns early with the hook reason surfaced to the user); `allow` + `additional_context` injects extra context the model sees this turn; `hook_specific_output.tool_input` is unused. This is the highest-value new event (input validation, prompt-prefix injection, secret-scanning).

4. PreCompact (effort S) — fires inside `_run_compaction` (agent_loop.py:941-982) just before `await self.compact()` (line 961). Payload: trigger ("auto"|"emergency"|"manual"), current_context_tokens, threshold. Handler: notification + optional `additional_context` mapped onto `compact()`'s `extra_instructions` param (agent_loop.py:2141) so a hook can steer the summary.

5. Stop (effort M) — fires when the assistant has finished and is about to yield control back to the user (the natural end of a user turn). Distinct from POST_AGENT_TURN: POST_AGENT_TURN already exists and can deny→retry; Stop is the Claude Code "the agent wants to stop" gate. To avoid duplicating POST_AGENT_TURN, model Stop as POST_AGENT_TURN's sibling fired at the SAME point (agent_loop.py:1179-1185) but AFTER post-agent-turn retries are exhausted, with payload including `stop_hook_active` (true if we are already inside a Stop-induced continuation) to prevent infinite loops. `decision: "deny"` + reason injects a continuation user message (same mechanism as `HookUserMessage`), capped by the existing `HookRetryState`. If POST_AGENT_TURN already covers the user's need, Stop is optional; ship it second.

6. Notification (effort S) — fires when vibe needs user attention: a tool is about to block on an approval prompt, or `ask_user_question` is invoked. Payload: notification_type ("permission_required"|"input_idle"|"question"), message, tool_name (optional). Notification only (cannot gate). Fire point: in the approval path (agent_loop.py around the `approval_callback` invocation, ~1558) and where `ask_user_question` resolves. Primary use: desktop/Slack pings via an external command. Low coupling, reuses `_team_lifecycle`-style passthrough handler.

Shared design: add ONE generic `NotificationHookHandler(HookHandler)` (clone of `TeamLifecycleHandler` semantics — allow→emit system_message, deny→non-binding warning, passthrough) and register SessionStart/SessionEnd/PreCompact/Notification to it. UserPromptSubmit gets a new `UserPromptSubmitHandler` (gate + context injection). Stop reuses/subclasses `PostAgentTurnHandler` (retry-on-deny). Extend `match` validity: `match` and `strict` stay tool-only (models.py:62-70) — add SessionStart/UserPromptSubmit/etc to the "match not allowed" set.

PART B — Plugin/bundle manifest (effort L). A plugin is a directory with a `plugin.toml` (or `vibe-plugin.toml`) manifest at its root that declares component subdirectories and a hooks file. New config key `plugin_paths: list[Path]` (WithConcatMerge, alongside agent_paths at vibe_schema.py:127) plus a discovery dir `~/.vibe/plugins/*` and `<root>/.vibe/plugins/*`. At config-load time (after settings merge, before managers are built), each plugin manifest is resolved into the EXISTING path lists: a loader reads `plugin.toml`, resolves its declared `agents`/`skills`/`tools`/`workflows`/`prompts` dirs relative to the plugin root, and APPENDS them to `config.agent_paths`/`skill_paths`/`tool_paths`/`workflow_paths`; resolves `mcp_servers` (inline table array, same shape as MCPServer) and UNION-merges them into `config.mcp_servers`; resolves `hooks` (a path to a hooks.toml, or inline `[[hooks]]`) and feeds them into the same `HookConfigResult` that `load_hooks_from_fs` produces. This is purely additive layering — no manager touches plugins directly; they only ever see the expanded path lists / mcp list / hook list. A plugin can be `enabled`/`disabled` by name via new `enabled_plugins`/`disabled_plugins` glob lists (mirrors enabled_agents at vibe_schema.py:134). Plugins are gated behind `enable_experimental_hooks` (or a new `enable_experimental_plugins`) so the surface stays opt-in.

Manifest schema (`plugin.toml`): `name` (str, required, unique), `version` (str, optional), `description` (str, optional), `agents`/`skills`/`tools`/`workflows`/`prompts` (str|list[str], dirs relative to plugin root, default conventional subdir names if omitted), `mcp_servers` (array of MCPServer tables), `hooks` (str path to hooks.toml OR inline `[[hooks]]` array). Loader output: a `PluginLoadResult` mirroring `HookConfigResult` (`plugins`, `expanded_paths`, `hooks`, `mcp_servers`, `issues`) so malformed manifests degrade to warnings, never crash startup.

## Config

- `enable_experimental_hooks` (bool, default `false`) — Existing gate (vibe_schema.py:231); new hook events stay behind it so the surface remains opt-in.
- `plugin_paths` (list[Path], default `[]`) — Explicit plugin root directories or manifest files, merged WithConcatMerge like agent_paths.
- `enabled_plugins` (list[str], default `[]`) — Glob/regex allowlist of plugin names; if set, only these plugins load (mirrors enabled_agents).
- `disabled_plugins` (list[str], default `[]`) — Glob/regex denylist of plugin names; ignored when enabled_plugins is set.
- `enable_experimental_plugins` (bool, default `false`) — Optional separate gate for the plugin loader; can fold into enable_experimental_hooks to reduce flags.
- `[hooks].type (in hooks.toml)` (str enum, default `(required)`) — Extended HookType enum now accepts session_start|session_end|user_prompt_submit|pre_compact|stop|notification.
- `plugin.toml: name` (str, default `(required)`) — Unique plugin identifier used for enabled/disabled matching and duplicate detection.
- `plugin.toml: agents|skills|tools|workflows|prompts` (str|list[str], default `conventional subdir names`) — Component dirs (relative to plugin root) appended to the matching config *_paths.
- `plugin.toml: mcp_servers` (array of MCPServer tables, default `[]`) — MCP servers union-merged into config.mcp_servers by name.
- `plugin.toml: hooks` (str path | inline [[hooks]], default `plugin_root/.vibe/hooks.toml`) — Hook definitions folded into the shared HookConfigResult.

## Edge cases
- UserPromptSubmit block must still produce a coherent transcript: the user message was already appended (agent_loop.py:1095) and UserMessageEvent yielded, so blocking has to surface the reason without leaving a dangling user turn — append a synthetic assistant/system reply or pop the user message.
- Stop infinite loop: a Stop hook that always denies would loop forever; guard with stop_hook_active in the payload AND the existing HookRetryState cap (_MAX_RETRIES=3).
- Stop vs POST_AGENT_TURN double-injection: both fire at 1179-1185; ensure Stop fires only after post-agent-turn retry is None, and that a Stop continuation re-enters the loop cleanly.
- SessionEnd at process exit: parent may be tearing down the event loop; a slow/hung SessionEnd hook must not deadlock shutdown — bound it and swallow timeouts.
- SessionStart additional_context injection must happen before the first LLM turn but not double-count tokens against auto-compact thresholds unexpectedly.
- PreCompact runs inside the emergency-compaction recovery path (agent_loop.py:1148); a PreCompact hook that itself raises must not abort the recovery — treat failures as passthrough.
- Plugin name collisions across two plugins, or a plugin name colliding with a top-level hook name, must degrade to an issue (not a crash) like the existing duplicate-hook guard.
- Untrusted project roots: plugin discovery under <root>/.vibe/plugins must respect the same trust gating as hook_files (project source only when trusted, _harness_manager.py:39-46).
- Relative component dirs in plugin.toml that escape the plugin root (e.g. ../../) should be rejected to avoid arbitrary path injection.
- A plugin declaring mcp_servers with a name already in config.mcp_servers: union-merge semantics must match WithUnionMerge(merge_key=name) so user config wins / merges predictably.
- disabled_plugins must still allow the manifest to be read for its name (so it can be matched) but skip applying any of its components.
- Hooks delivered by a plugin must be subject to the same enable_experimental_hooks gate; loading plugin hooks while hooks are globally disabled would be surprising.

## Test plan
- Unit: build_invocation returns the right subclass for each of the 6 new HookType values and raises on missing required fields (mirror existing models tests).
- Unit: HookConfig rejects match/strict for session_start/user_prompt_submit/etc (extend the POST_AGENT_TURN constraint test).
- Unit: NotificationHookHandler — allow emits system_message, deny is non-binding warning + passthrough, empty stdout is passthrough.
- Unit: UserPromptSubmitHandler — deny yields a block decision; allow with additional_context yields injectable context; retry cap respected.
- Integration: a user_prompt_submit hook returning deny causes _conversation_loop to return without an LLM call and surfaces the reason; allow+context injects a message the model sees.
- Integration: a pre_compact hook's additional_context reaches compact(extra_instructions=...).
- Integration: a stop hook denying once injects a continuation and re-enters the loop; denying past the cap stops; stop_hook_active prevents runaway.
- Integration: session_start/session_end fire exactly once per run in programmatic mode and once each across a /clear via _reset_session.
- Integration: a hung session_end hook does not block process exit (timeout path).
- Plugin loader: a valid plugin.toml expands agent_paths/skill_paths/tool_paths/workflow_paths, union-merges mcp_servers, and folds hooks into HookConfigResult.
- Plugin loader: malformed manifest, name collision, disabled-by-glob, and path-escape cases each produce an issue and do not crash startup.
- Plugin loader: enabled_plugins allowlist restricts which plugins load; trust gating skips untrusted project plugin dirs.
- E2E: drop a sample plugin under ~/.vibe/plugins and confirm its agent, skill, tool, MCP server, and hook all become active in one session.

## Risks
- Stop + POST_AGENT_TURN overlap risks confusing/duplicated retry behavior; if POST_AGENT_TURN already satisfies users, Stop adds surface for little gain — ship it last or fold into POST_AGENT_TURN.
- Firing hooks at process-exit (SessionEnd) introduces a shutdown-latency and deadlock surface; must be strictly time-bounded.
- UserPromptSubmit blocking mid-turn touches the carefully-ordered message/injection bookkeeping in _conversation_loop (1095-1195, including the finally drain); a wrong edit can lose or duplicate messages.
- The plugin loader runs at startup before managers are built; an exception there would break ALL launches — it must be fully defensive (issues, never raises), like load_hooks_from_fs.
- Plugins that bundle MCP servers + tools effectively let a manifest run arbitrary code/processes; trust gating and the experimental flag are the only guardrails — security review needed.
- Three call sites (cli, acp x2, programmatic) must all wire the new loader consistently or behavior diverges between entrypoints — same divergence risk already exists for hooks.
- Adding 6 enum members touches build_invocation's assert_never exhaustiveness and any other match over HookType; missing a case is a latent runtime error.
- Config merge ordering: plugin-expanded paths must be appended AFTER user/project config merge so user config can't be silently overridden, and so WithConcatMerge order stays deterministic.

Generalizing to 'reject for all non-tool types' is a behavior change for the existing team-lifecycle types, not just the new ones.
- Stop 'fired at the SAME point 1179-1185 after post-agent-turn retries' assumes post-turn retry exhaustion is observable there. But _dispatch_post_turn_hooks returns retry_msg=None both when a hook allowed AND when retries are exhausted — the loop cannot distinguish 'no deny' from 'deny but capped' at 1179-1185 without new signaling, so 'fire Stop only after post-agent-turn retry is None' is ambiguous.

**Missing pieces:**
- Two parallel config models must BOTH be edited for every new key: VibeConfigSchema (vibe_schema.py, the merge layer with WithConcatMerge annotations) AND VibeConfig (_settings.py, the runtime BaseSettings model with _expand_* validators). The spec mentions both files but under-states that plugin_paths/enabled_plugins/disabled_plugins/enable_experimental_plugins each need duplicated definitions kept in sync, plus a _expand_plugin_paths validator mirroring _expand_tool_paths/_expand_skill_paths/_expand_workflow_paths (1115-1134).
- UserPromptSubmit 'block→return' edge: the user message is appended at 1089-1095 and reset_retry_count runs at 1109-1110, all BEFORE the proposed fire point. The finally block at 1187-1195 drains pending injections and saves. A clean block must pop the dangling user message (or append a synthetic assistant reply) AND still let the finally save a coherent transcript — the spec flags this as an edge case but provides no concrete mechanism, and there is no existing 'early-return-with-reason' path in _conversation_loop to copy.
- No existing value object for a UserPromptSubmit block: HookUserMessage (retry inject) and HookToolDenial (tool error) exist, but neither models 'block this prompt + surface reason to user'. A new decision type + a new _HookYield union member (_handler.py:28-34) + a new event for the UI must be added; the spec hand-waves 'HookPromptBlock / reuse HookToolDenial-style'.
- Notification fire point requires plumbing the hook into the approval/user-input CALLBACK layer (InvokeContext.approval_callback / user_input_callback / _ask_approval at 1724), not the agent loop. That means either wrapping the callbacks in AgentLoop before they reach InvokeContext, or firing inside ask_user_question / approval tools — a cross-cutting change the spec does not scope.
- Part B: no PluginManifest / PluginLoadResult / load_plugins_from_fs module exists (vibe/core/plugins/ does not exist). Also missing: how plugin hooks respect that load_hooks_from_fs short-circuits and returns empty when enable_experimental_hooks is False (config.py:78-79) — plugin hooks must be folded AFTER that gate or they'd bypass it.
- Part B trust gating: harness_files _trusted_workdir (39-46) is the existing mechanism, but plugin_dirs would need to replicate the project-source trust check; the spec names this as an edge case but the loader must actually call trusted_folders_manager.is_trusted per project root, which is internal to HarnessFilesManager today.
- SessionStart additional_context injection target: spec says append to _pending_injected_messages, but those are drained at the TOP of the loop's first turn only if staged before act()/_conversation_loop runs; SessionStart fired in __init__/initialize path (516) runs before any _conversation_loop exists, so the injection timing vs first-turn drain needs explicit wiring.
